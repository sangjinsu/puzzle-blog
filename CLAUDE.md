# 🧩 Puzzle Content Lab

> 퍼즐 게임 데모를 통해 배우는 게임 콘텐츠 서버 설계 블로그

---

## 프로젝트 개요

퍼즐 게임(3매치)의 **인터랙티브 데모**를 직접 플레이하면서, 그 뒤에서 동작하는 **서버 로직과 DB 설계**를 자연스럽게 학습하는 블로그입니다. 일반인도 이해할 수 있도록 **미니멀한 인포그래픽** 중심으로 구성합니다.

---

## 핵심 콘셉트

### 블로그 구조: 3 Layer

```
┌─────────────────────────────────────┐
│  🎮 PLAY    인터랙티브 데모 (3매치)    │
├─────────────────────────────────────┤
│  ⚙️ LOGIC   서버 & DB 로직 해설       │
├─────────────────────────────────────┤
│  📊 COMPARE  퍼즐 게임 콘텐츠 비교     │
└─────────────────────────────────────┘
```

- **PLAY**: 로얄매치 스타일 3매치 데모를 직접 플레이
- **LOGIC**: 각 콘텐츠(라이프, 퀘스트, 패스 등)의 서버/DB 구현을 인포그래픽으로 설명
- **COMPARE**: 로얄매치 · 캔디크러쉬 · 트리플타일 등 실제 게임의 콘텐츠 비교 분석

### 홈페이지 구성

홈페이지(`/`)는 두 가지 핵심 요소로 구성합니다:

1. **데모 소개** — 3매치 퍼즐 데모의 하이라이트 및 플레이 진입점
2. **블로그 설명** — Puzzle Content Lab의 목적, 학습 방식, 콘텐츠 구조 안내

## 3매치 데모 스펙

### 매칭 룰 (로얄매치 기반)

```
기본 매치 (3개 이상 일직선)
■ ■ ■        ■
             ■
             ■

T매치 / L매치 (5개)
■ ■ ■        ■ ■ ■
  ■            ■
  ■              ■

2×2 스퀘어 매치
■ ■
■ ■
```

### 아이템 생성 조건

| 아이템 | 생성 조건 | 효과 |
|--------|----------|------|
| 🚀 가로 로켓 | 4매치 (세로) | 가로 한 줄 제거 |
| 🚀 세로 로켓 | 4매치 (가로) | 세로 한 줄 제거 |
| 💣 폭탄 | 2×2 매치 또는 T/L 매치 | 주변 3×3 범위 제거 |

### 데모 구현 우선순위

1. **타일 시스템**: 그리드 생성, 타일 타입, 렌더링
2. **매칭 로직**: 3매치 · 4매치 · 2×2 · T/L 감지
3. **물리 시스템**: 타일 낙하, 빈칸 채우기
4. **아이템 로직**: 생성 조건 판별, 아이템 발동 효과
5. **애니메이션**: 스왑, 제거, 낙하 트랜지션
6. **스테이지**: 목표 시스템 (특정 타일 N개 제거 등)

## 디자인 원칙

### 미니멀 & 인포그래픽

```
DO                              DON'T
─────────────────────────────────────────
✓ 깔끔한 다이어그램             ✗ 장황한 텍스트 블록
✓ 단계별 시각 플로우            ✗ 코드만 나열
✓ 비교 차트/매트릭스            ✗ 표 없는 나열식 비교
✓ 인터랙티브 데모 우선          ✗ 스크린샷만 나열
✓ 핵심 개념 → 서버 흐름 → 코드  ✗ 코드부터 시작
```

### 인포그래픽 우선 원칙

**동작/행위 설명은 반드시 인포그래픽으로 표현합니다.** 텍스트만으로 동작을 서술하지 않습니다.

| 설명 대상 | 인포그래픽 유형 |
|-----------|----------------|
| API 요청/응답 흐름 | 시퀀스 다이어그램 |
| 상태 전이 (라이프, 세션 등) | 상태 다이어그램 |
| 데이터 처리 절차 | 플로우차트 |
| DB 구조 & 관계 | ER 다이어그램 |
| 시간 기반 이벤트 | 타임라인 |
| 게임 간 비교 | 비교 매트릭스/차트 |

- 포스트 당 최소 4개 이상의 인포그래픽 포함
- 서버 동작 흐름은 시퀀스 다이어그램 필수
- 모든 인포그래픽은 SVG + Framer Motion 애니메이션 적용

## 프로젝트 구조 (FSD — Feature-Sliced Design)

```
src/
├── app/                        # [App Layer] Next.js App Router
│   ├── layout.tsx
│   ├── page.tsx                # 홈: 데모 소개 + 블로그 설명
│   ├── posts/[slug]/page.tsx
│   └── demo/page.tsx
├── pages/                      # [Pages Layer] 페이지 컴포지션
│   ├── home/ui/
│   ├── post-detail/ui/
│   └── demo/ui/
├── widgets/                    # [Widgets Layer] 독립 UI 블록
│   ├── puzzle-board/{ui,model}/
│   ├── infographic-viewer/ui/  # FlowDiagram, CompareChart, SequenceDiagram 등
│   ├── post-list/ui/
│   └── code-viewer/ui/         # Go/Java/Python/TS 탭 코드 블록
├── features/                   # [Features Layer] 비즈니스 기능
│   ├── tile-swap/{ui,model}/
│   ├── match-detection/model/
│   ├── gravity/model/
│   ├── item-generation/model/
│   └── stage-clear/{ui,model}/
├── entities/                   # [Entities Layer] 도메인 객체
│   ├── tile/{ui,model,lib}/
│   ├── stage/model/
│   └── post/{ui,model}/
├── shared/                     # [Shared Layer] 공통 모듈
│   ├── ui/
│   ├── lib/
│   ├── config/
│   └── assets/
└── content/                    # MDX 포스트 (FSD 외부)
```

### FSD 레이어 의존성 규칙

```
app → pages → widgets → features → entities → shared
          (상위 레이어는 하위 레이어만 import 가능)
```

| 레이어 | 역할 | 예시 |
|--------|------|------|
| **app** | Next.js 라우팅, 글로벌 프로바이더 | `layout.tsx`, `page.tsx` |
| **pages** | 페이지 단위 컴포지션 | 위젯 조합, 데이터 페칭 |
| **widgets** | 독립적으로 동작하는 UI 블록 | 퍼즐 보드, 인포그래픽 뷰어, 코드 뷰어 |
| **features** | 사용자 시나리오 단위 기능 | 타일 스왑, 매치 감지, 스테이지 클리어 |
| **entities** | 비즈니스 도메인 객체 | Tile, Stage, Post |
| **shared** | 재사용 유틸, UI 킷, 상수 | 색상, 타입, 공통 컴포넌트 |

## 핵심 타입 정의 (참고)

```typescript
// 타일 타입
type TileColor = 'red' | 'blue' | 'green' | 'yellow' | 'purple';

type TileType = 'normal' | 'rocket_h' | 'rocket_v' | 'bomb';

interface Tile {
  id: string;
  color: TileColor;
  type: TileType;
  row: number;
  col: number;
}

// 매치 결과
interface MatchResult {
  tiles: Tile[];
  pattern: 'three' | 'four_h' | 'four_v' | 'square' | 'T' | 'L';
  itemGenerated?: TileType;
  itemPosition?: { row: number; col: number };
}

// 스테이지
interface Stage {
  id: number;
  gridSize: { rows: number; cols: number };
  moves: number;
  objectives: Objective[];
}

interface Objective {
  type: 'collect_color' | 'collect_item' | 'clear_obstacle';
  target: string;
  count: number;
}
```

## 작성 가이드라인

### 포스트 구조 (각 글)

```
1. 🎯 한 줄 요약 — 이 콘텐츠가 뭔지
2. 🎮 데모 — 직접 플레이/인터랙션
3. 📐 개념 설명 — 인포그래픽 다이어그램
4. ⚙️ 서버 설계 — API 흐름 + DB 스키마
5. 💻 핵심 코드 — 서버 코드 스니펫 (Go, Java, Python, TypeScript)
6. 📊 게임 비교 — 3~4개 게임의 해당 콘텐츠 비교
7. 💡 정리 — 핵심 포인트 요약
```

### 톤 & 매너

- **대상**: 게임 개발에 관심 있는 일반인 ~ 주니어 개발자
- **문체**: 친근하고 명확한 설명체 (~입니다)
- **분량**: 포스트 당 읽기 5~8분
- **비주얼**: 텍스트 < 다이어그램, 최소 4개 인포그래픽/포스트
- **인포그래픽 필수**: 동작/행위 설명은 반드시 인포그래픽으로 표현 (텍스트만 서술 금지)

## 참고

- 서버 코드 예시는 **Go, Java, Python, TypeScript** 4개 언어로 작성 (탭 전환 코드 블록)
- 데모 클라이언트는 **Next.js + TypeScript**
- 데모는 서버 로직을 시각적으로 보여주는 도구이지, 주인공이 아님
- DB 설계는 개념적으로 설명 — 특정 구현체에 종속하지 않음
- 블로그 콘텐츠 로드맵은 예시이며, 실제 포스트는 추후 추가
